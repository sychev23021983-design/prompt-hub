# prompt-hub

Единый дашборд по трём SEO-проектам (shustrik-maps.com, itc.by, fit.shustrik-maps.com):
таблица страниц/карточек со статусом, кнопка "Промпт" генерирует готовый текст для LLM
на основе данных строки (свой стиль промпта под каждый сайт), кнопка статуса отмечает
прогресс.

## Что внутри
- `app/server.py` — Flask-приложение (роуты, БД, генерация промптов)
- `app/db.py` — схема SQLite
- `app/import.py` — импорт из трёх xlsx в БД (сохраняет прогресс Applied? при повторном импорте)
- `app/prompts.py` — логика данных (строка → контекст промпта, поиск связанных
  страниц, тип страницы); сам текст промпта рендерится из файлов ниже
- `app/structure.py` — дерево структуры сайта (парсинг вставленного текста,
  слаги, сортировка, подсказка URL) — см. раздел "Структура сайта" ниже
- `app/prompt_config/` — конфиг стиля/лимитов под каждый сайт (`commerce_en.yaml`
  — shustrik-maps, английский; `b2b_ru.yaml` — itc.by; `personal_ru.yaml` —
  fit.shustrik-maps) + общие значения по умолчанию (`_shared_defaults.yaml`,
  `_shared_ai_openers.yaml`) и `CHANGELOG.md`
- `app/prompt_templates/` — Jinja2-шаблон итогового промпта (`master_prompt.md.j2`,
  общий для всех сайтов) и общий чек-лист самопроверки Yoast (`_shared/quality_check.md.j2`)
- `app/templates/index.html` — дашборд
- `import_data/` — исходные xlsx (кладите сюда актуальные версии с каждого проекта)
- `data/` — SQLite база (создаётся автоматически, переживает пересоздание контейнера)

### Как редактировать стиль/тон/лимиты промптов
Не нужно трогать Python-код. Правьте файлы в `app/prompt_config/` (персона,
банворды, тон, лимиты символов/слов, FAQ/картинки) или `app/prompt_templates/`
(формат вывода, чек-лист) → `git add . && git commit && git push` → на сервере
`git pull` (пересборка контейнера не нужна, `app/` смонтирован как volume).
История изменений — в `app/prompt_config/CHANGELOG.md`.

## Деплой на сервере (под ваш процесс)

```bash
cd /root/prompt-hub
git pull
docker compose up -d --force-recreate
```

**Важно про кэш:** код (`app/`) и шаблоны смонтированы как volume, а не запечены в
образ — поэтому `--force-recreate` без `--build` всё равно подхватывает свежий код.
Пересборка образа (`docker compose build`) нужна только если менялся
`app/requirements.txt` (новая зависимость).

Первый запуск:
```bash
cd /root/prompt-hub
docker compose up -d --build
```
При первом старте контейнер автоматически создаст базу и импортирует все три xlsx
из `import_data/`.

Приложение слушает **порт 8040** (см. `docker-compose.yml`).

## Как обновлять xlsx-данные

1. Обновили xlsx одного из проектов (после новой SEO-сессии) → положите свежий файл
   в `import_data/`, заменив старый (имя файла должно совпадать, см. `app/import.py`,
   словарь `SITES`)
2. `git add . && git commit -m "update data" && git push`
3. На сервере: `git pull`, затем нажмите кнопку **"Обновить данные из xlsx"** в самом
   дашборде (не нужно перезапускать контейнер) — либо `docker compose restart prompt-hub`
4. Прогресс (Applied?/Notes) сохраняется по ключу URL/название — повторный импорт его
   не затирает

## Кнопка "Экспорт {sайт}.xlsx"

Выгружает текущее состояние таблицы (включая отметки Applied?) обратно в xlsx — чтобы
можно было закоммитить актуальный прогресс в соответствующий репозиторий проекта
(`shustrik-maps-seo`, `itc-by-seo`, `fit-shustrik-maps-seo`) вручную, как вы делали раньше.

## Структура сайта

У каждого сайта есть вкладка **"Структура"** (кнопка "📂 Структура сайта" появляется,
когда в основной таблице выбран конкретный сайт): раскрывающееся дерево разделов,
которое вы заполняете вручную — обычно вставкой из KeyCollector.

- **Вставка дерева**: вставьте в текстовое поле список с отступами (табы или
  пробелы = уровень вложенности), нажмите "Импортировать/обновить". Повторная
  вставка обновлённого списка ничего не удаляет и не рвёт уже сделанные
  привязки — совпадающие по названию и родителю узлы не дублируются, новые
  добавляются.
- **Привязка страниц**: в основной таблице появилась колонка "Структура" —
  выпадающий список со всеми разделами дерева (с отступом по уровню). После
  привязки таблица автоматически сортируется в порядке дерева (непривязанные
  страницы — общим хвостом в конце).
- **URL**: колонка URL стала редактируемой (как остальные поля). Кнопка 🔗
  рядом с полем предлагает URL на основе пути страницы в дереве (слаги
  разделов) + слага её названия — можно принять как есть или поправить.
- **Слаги разделов** редактируются прямо в дереве (клик по сегменту рядом с
  названием) — от них зависит предлагаемый URL.
- Узел можно удалить только если в нём нет ни подразделов, ни привязанных
  страниц — это защита от случайной потери данных.
- Привязка к структуре и вручную заданный URL **переживают** переимпорт xlsx
  (кнопка "Обновить данные из xlsx") — они сопоставляются по названию строки,
  как и Applied?/Notes.

## Промпт-шаблоны (кастомные шаблоны по типу страницы)

У разных типов страниц (лендинг, о нас, страница доверия, FAQ, услуга, блог)
нужна разная структура промпта — не только данные, но и логика. На вкладке
**"🧩 Промпт-шаблоны"** (кнопка в тулбаре, доступна всегда) можно сохранить
кастомный Jinja2-шаблон промпта для конкретной пары "сайт + тип страницы".

- В основной таблице у каждой строки появилась колонка **"Тип страницы"** —
  выпадающий список (лендинг / о нас / доверие / FAQ / услуга / блог).
- На вкладке шаблонов — матрица "тип страницы × сайт": клик открывает редактор
  с текстовым полем, кнопками "Сохранить" и "Превью на первой строке сайта"
  (рендерит шаблон на реальных данных, не сохраняя — удобно для проверки
  синтаксиса перед сохранением).
- **Приоритет генерации** (кнопка "Промпт" в основной таблице): если у строки
  выбран тип страницы и для (сайт, тип) сохранён кастомный шаблон — используется
  он; иначе работает обычный движок (`prompt_config/*.yaml` + `master_prompt.md.j2`)
  как раньше.
- Кастомному шаблону доступны те же данные, что и обычному движку: `row.*`
  (любое поле строки напрямую), `breadcrumb`, `related_block`/`has_related`,
  `input_lines`, `cfg.*` (персона/тон/банворды сайта), `limits.*` — подробный
  список переменных с примерами есть прямо на вкладке шаблонов.
- Тип страницы, как и URL и привязка к структуре, **переживает** переимпорт xlsx.

## Трёхэтапная генерация (план → текст → изображения)

Необязательный режим поверх обычного промпта — включается сам по себе, если
вы им пользуетесь, и никак не мешает, если нет.

- **1️⃣ План** — компактный промпт: определить интент (основной/вторичный),
  коммерческую стадию, тип услуги, подход к подаче, структуру блоков, какие
  FAQ и связанные страницы использовать. Результат вставляется в поле
  "План страницы" (оно редактируемое, как Notes).
- **2️⃣ Текст** — обычная кнопка "Промпт". Если поле "План страницы" заполнено,
  промпт автоматически подставляет его целиком и просит следовать ему, вместо
  того чтобы модель заново определяла интент/тип/подход. Если поле пустое —
  работает как раньше, полностью самостоятельно.
- **3️⃣ Картинки** — отдельный промпт только на 5 изображений (ALT/Title/Caption/
  место на странице). Если план заполнен — учитывает его для соответствия тону/структуре.

Как и с LSI — это не автогенерация через API, а подготовка промпта для
копирования в ChatGPT/Claude, результат вставляется вручную обратно в поле.

## Убрана "Самопроверка по Yoast"

Раздел с таблицей самопроверки по критериям Yoast убран из промпта — эти
проверки и так делает сам плагин Yoast в WordPress, повторный запрос к модели
их продублировать только раздувал промпт. Осталась только проверка на
стоп-слова/AI-штампы (генерируется из banned_words/ai_openers конфига сайта) —
она специфична для этого движка и Yoast её не покрывает.

## Безопасность



Приложение не имеет доступа к GitHub API и не хранит никаких токенов — весь git
pull/push вы делаете сами на сервере через уже настроенный у вас доступ. Открыто без
авторизации (просто по адресу `http://147.45.42.169:8040`) — если нужен пароль на
вход, добавьте basic-auth на уровне nginx/Caddy перед этим сервисом, либо скажите,
добавлю Flask-Login на следующем шаге.
